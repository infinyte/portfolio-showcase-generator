## Getting Started

### 1. Fork the Repository

Click the "Fork" button in the top right of the repository page.

### 2. Clone Your Fork

```bash
git clone https://github.com/YOUR_USERNAME/portfolio-showcase-generator.git
cd portfolio-showcase-generator
```

### 3. Create a Branch

```bash
git checkout -b feature/your-feature-name
```

### 4. Make Your Changes

- Follow the existing style and structure
- Test your changes with Claude
- Update documentation if needed

### 5. Commit Your Changes

```bash
git add .
git commit -m "Add: Brief description of your changes"
```

Use conventional commit messages:
- `Add:` for new features
- `Fix:` for bug fixes
- `Docs:` for documentation changes
- `Example:` for adding examples
- `Refactor:` for code improvements

### 6. Push to Your Fork

```bash
git push origin feature/your-feature-name
```

### 7. Create Pull Request

- Go to your fork on GitHub
- Click "Pull Request"
- Provide clear description of changes
- Reference any related issues

## Contribution Guidelines

### Code Quality

- **Clear and readable** - Others should understand your changes
- **Well-documented** - Add comments for complex logic
- **Tested** - Verify changes work with Claude
- **Consistent style** - Follow existing patterns

### Documentation

- **Clear language** - Avoid jargon when possible
- **Good examples** - Show, don't just tell
- **Proper formatting** - Use markdown correctly
- **Updated TOCs** - Keep tables of contents current

### Examples

When adding example outputs:
- **Anonymize sensitive data** - Remove company names, real metrics if needed
- **Show real value** - Use realistic project descriptions
- **Demonstrate variety** - Different project types and use cases
- **Include metadata** - What was the input? What outputs were generated?

### Skill Improvements

When modifying SKILL.md:
- **Test thoroughly** - Ensure changes work as expected
- **Document changes** - Update relevant documentation
- **Backward compatible** - Don't break existing functionality
- **Clear rationale** - Explain why the change is beneficial

## Review Process

### What to Expect

1. **Initial Review** - Maintainer reviews within 1-2 weeks
2. **Feedback** - May request changes or clarifications
3. **Iteration** - Work together to refine contribution
4. **Approval** - Once ready, PR will be merged
5. **Recognition** - Contributors added to CONTRIBUTORS.md

### Review Criteria

- **Quality** - Does it improve the project?
- **Compatibility** - Does it work with existing features?
- **Documentation** - Is it well-explained?
- **Testing** - Has it been verified?
- **Value** - Does it benefit users?

## Community Guidelines

### Be Respectful

- **Constructive feedback** - Focus on improvement, not criticism
